# Storage Persistente — 2026-05-22

## Contesto

Il progetto è un gioco Pokémon-like compilato a WASM e servito come SPA statica.
Serve persistere: PlayerState, run progress, team, cache risposte API PokéAPI.

## Opzioni valutate

### SQLite WASM (sqlite3.wasm ufficiale)
- Richiede `SharedArrayBuffer` per modalità OPFS multi-thread
- `SharedArrayBuffer` richiede headers COOP/COEP (`Cross-Origin-Opener-Policy: same-origin`)
- Questi headers rompono embedding in iframe e OAuth redirect su mobile
- La modalità single-thread (senza SAB) funziona ma usa un VFS in-memory o OPFS-sync,
  entrambi sperimentali nel 2026 su Safari iOS

**Scartato**: complessità deploy, compatibilità mobile incerta.

### OPFS diretto (File System Access API)
- Stesso problema: richiede COOP/COEP su Safari per accesso sincrono
- API asincrona disponibile ma non supportata uniformemente su mobile

**Scartato**: stessa limitazione di deploy.

### IndexedDB via `rexie`
- `rexie` è un wrapper ergonomico su IndexedDB per Rust/WASM
- IndexedDB è supportato su tutti i browser moderni incluso Safari iOS 8+
- Nessun header speciale richiesto
- API asincrona, compatibile con `spawn_local`
- Supporta strutture JSON serializzate con `serde_wasm_bindgen`

**Scelto** come backend primario.

## Architettura

```
frontend/src/core/storage/
├── mod.rs          — trait StorageBackend + tipo Store (context)
├── idb.rs          — implementazione IndexedDB via rexie
└── models.rs       — struct serializzabili (PlayerState, RunProgress, TeamEntry, CachedPokemon)
```

### Trait astratto

```rust
pub trait StorageBackend {
    async fn save(&self, store: &str, key: &str, value: &JsValue) -> Result<(), String>;
    async fn load(&self, store: &str, key: &str) -> Result<Option<JsValue>, String>;
    async fn delete(&self, store: &str, key: &str) -> Result<(), String>;
}
```

L'astrazione permette di aggiungere un backend OPFS/SQLite in futuro senza toccare i chiamanti.

### Object stores IndexedDB

| Store          | Chiave        | Contenuto                        |
|----------------|---------------|----------------------------------|
| `player_state` | `"current"`   | `PlayerState` (nome, money, ecc) |
| `run_progress` | `"current"`   | `RunProgress` (badge, step)      |
| `team`         | chiave slot   | `TeamEntry` (pokemon serializzato)|
| `api_cache`    | nome pokemon  | `PokemonData` JSON               |

### Context Leptos

`Store` wrappa `Rc<IdbBackend>` ed è fornito via `provide_context` in `App`,
esattamente come `PokemonCache`. I componenti usano `use_storage()`.

## Note

- `rexie` gestisce internamente la migrazione dello schema (versioning)
- La cache API esistente (`PokemonCache` in memoria) resta invariata;
  `api_cache` IDB viene letta all'avvio e scritta ad ogni fetch per sopravvivere al reload
