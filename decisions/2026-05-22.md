# 2026-05-22

## Struttura workspace monorepo (game-core + frontend)

**Contesto:** Progetto Pokémon-like in Rust + Leptos + WASM da zero. Necessità di separare logica di gioco dalla UI.

**Motivazione:** Tenere la logica pura in `game-core` (senza dipendenze WASM) permette di eseguire `cargo test` velocemente senza compilare WASM. La logica rimane riusabile e testabile in isolamento. `frontend` dipende da `game-core` e gestisce solo presentazione.

**Conseguenze:** Ogni feature di gioco va prima implementata e testata in `game-core`, poi esposta al frontend. Nessuna logica di battaglia o stato di gioco nel crate `frontend`.

## Tipi base: Type, Move, Pokemon con formula HP Gen III+

**Contesto:** Step 1 dello scaffold — definire i tipi fondamentali del dominio.

**Motivazione:** `Type` come enum esaustivo copre tutte le 18 tipologie Gen VI+. `type_effectiveness` con match pattern è esplicita, leggibile e non richiede tabelle esterne. Formula HP `floor((2*base+31)*level/100) + level + 10` usa IV fisso a 31 (massimo) per semplicità — si può parametrizzare in seguito. `Pokemon.take_damage` usa `saturating_sub` per evitare underflow senza gestione errori esplicita.

**Conseguenze:** IV fisso a 31 significa che i Pokémon hanno sempre HP massimi — da rivedere quando si introduce il sistema di cattura/allevamento. La tabella efficacia tipo va aggiornata se si aggiungono meccaniche speciali (es. Freeze-Dry).

## Frontend Leptos 0.7 CSR con leptos_router

**Contesto:** Setup del crate `frontend` con Leptos client-side rendering, routing `/home` e `/ui-kit`, struttura `components/`, `pages/`, `core/`.

**Motivazione:** Leptos 0.7 CSR (feature `csr`) è il target naturale per WASM puro senza SSR. `leptos_router` 0.7 non richiede feature aggiuntive per il browser — la feature `browser` non esiste, il router funziona di default in CSR. `Trunk` gestisce il bundle WASM + HTML. Il `[profile.release]` con `opt-level = "z"` e `lto = true` è nel workspace root per ridurre la dimensione del bundle WASM.

**Conseguenze:** Il progetto si avvia con `trunk serve` dalla directory `frontend/`. Le route non gestite mostrano un 404 inline. `components/` e `core/` sono vuoti per ora — verranno popolati incrementalmente.

## CSS: un file per componente, naming BEM, entry point globale

**Contesto:** Setup stile per i componenti UI della pagina /ui-kit. Valutate: CSS Modules (non supportati nativamente in Leptos), Stylist (scoped CSS a runtime), Tailwind, CSS plain per file.

**Motivazione:** CSS plain con naming BEM è la scelta a zero dipendenze. Ogni componente ha il suo file (es. `pokemon_card.css`) sotto `frontend/assets/styles/`. Un `style.css` centrale li importa tutti con `@import` — Trunk lo inietta automaticamente nell'HTML tramite `<link data-trunk rel="css">`. Nessun tooling aggiuntivo, struttura chiara, scalabile per le dimensioni del progetto.

**Conseguenze:** Le classi devono seguire la convenzione BEM (`.pokemon-card__hp`, `.battle-button--disabled`). Niente scope automatico — naming disciplinato è l'unica protezione da collisioni.

## Testing strategy: unit test in-module + integration test in tests/

**Contesto:** Aggiunta test per PokemonState, damage logic e turn system in game-core.

**Motivazione:** Due livelli di test distinti per scopo diverso. I test unitari (`#[cfg(test)]` dentro il modulo) coprono invarianti interni — dettagli implementativi che possono cambiare. I test di integrazione in `game-core/tests/` testano l'API pubblica come la userebbe un consumatore esterno, senza accesso a privati. La cartella `tests/` è divisa in `core/` (stato Pokémon) e `battle/` (logica di battaglia) per separare le responsabilità. Nessun mock: tutto Rust puro, zero dipendenze esterne.

**Conseguenze:** Ogni nuovo modulo di logica va prima implementato in `game-core/src/`, poi testato a livello di integrazione in `tests/`. I test unitari in-module rimangono per invarianti interni (es. formula HP).

## Battle engine: funzioni pure + RNG deterministico con LCG

**Contesto:** Aggiunta crit chance, damage result strutturato, turn order per velocità. Vincolo esplicito: deterministico e testabile.

**Motivazione:** I critici richiedono casualità, ma "deterministico" e "testabile" impongono di non usare `rand` con stato globale. Soluzione: LCG (Linear Congruential Generator) minimale in `battle::rng` — il seed viene passato esternamente alle funzioni. Le funzioni rimangono pure: stesso seed → stesso risultato. Il seed avanza ad ogni uso e viene restituito, permettendo alla chiamante di tenerlo aggiornato. Crit chance base Gen I: 1/16 (6.25%). La formula danno Gen III+ esistente viene estesa con moltiplicatore crit ×1.5. `DamageResult` trasporta danno finale, flag crit e moltiplicatore efficacia — utile per la UI (animazioni, messaggi). `TurnOutcome` aggrega i due `DamageResult`.

**Conseguenze:** Il battle engine è completamente deterministico: dato seed + input → output sempre identico. La UI dovrà mantenere e passare il seed ad ogni turno. Non si usa nessun crate esterno — zero dipendenze aggiuntive.

## Data layer PokéAPI: fetch nel frontend WASM, cache in-memory con HashMap

**Contesto:** Integrazione PokéAPI senza server. Vincolo: no SQLite, no backend, solo browser.

**Motivazione:** `game-core` deve rimanere Rust puro senza dipendenze WASM — non può fare HTTP. La fetch va nel crate `frontend` con `gloo-net` (HTTP client WASM-compatible, già dipendenza transitiva di Leptos). La cache è un `HashMap<String, PokemonData>` wrappato in un `RwSignal` e fornito come contesto Leptos — vive per tutta la sessione, accessibile da qualsiasi componente. Struttura in `frontend/src/core/`: `api.rs` (fetch + tipi), `cache.rs` (contesto cache). Separazione netta da `battle` — l'engine non sa nulla di rete.

**Conseguenze:** La cache è in-memory: si svuota a ogni reload. Niente persistenza per ora. Il fallback in caso di errore rete restituisce `Err(String)` — la UI decide come gestirlo. I tipi `PokemonData` vivono nel frontend, non in `game-core` — se in futuro serve condivisione si crea un crate `shared-types`.
